# AIPU — 제품 기획 (v0.1 초안, 결정-강제형)

> **성격.** 이 문서는 백지 기획이 아니라 **결정을 강제하는 초안**이다. 각 갈림길에
> **추천 기본값**을 채워 두었으니, 오너는 원점에서 시작하지 말고 **확인/수정**만
> 하면 된다. 근거는 저장소에 이미 있는 것(검증된 RTL·FPGA 핏·프로비저너·API 껍데기·
> 텔레메트리 리드백)과 정직한 제약(실리콘 없음, FPGA는 슬라이스)에 둔다.
> 성능 수치는 전부 **[추정]** — 대역폭 루프라인 투영이며 실리콘 측정이 아니다.

## 0. 결정된 것

- **비치헤드 = 개인 파워유저 (챗 + 코드).** 혼자 쓰는 고성능 로컬 모델 박스.
  에어갭 조직(RAG·GUI·시각화)은 **후속 비치헤드로 연기.** 이 선택의 핵심 이점:
  **RAG·GUI를 지을 필요가 없다** — OpenAI 호환 API로 기존 챗/IDE 클라이언트
  생태계(Open WebUI, Continue.dev 등)를 그대로 붙이면 되므로, v0.1에 필요한
  신규 소프트웨어가 가장 적다.

## 1. v0.1의 정직한 정의 — "경험"과 "하드웨어 증명"의 분리

실리콘이 없고 FPGA는 컴팩트 슬라이스(VOCAB=256)라 **진짜 GLM-5.2 텍스트를 못 낸다.**
그래서 v0.1은 두 갈래를 **분리해서 각각 정직하게** 증명한다:

| 갈래 | 무엇을 증명 | 어떻게 | 지금 가능? |
|---|---|---|---|
| **제품 경험** | 실제 GLM-5.2 텍스트로 챗·코딩이 오프라인에서 된다 | OpenAI 호환 API + **소프트웨어 풀모델 백엔드**(CPU/GPU, 정확하지만 느림), 표준 클라이언트로 | ✅ $0, 지금 |
| **하드웨어** | 우리 데이터패스가 실 칩에 들어가 실 사이클로 돈다 | FPGA 슬라이스(실 P&R 46.5MHz, 토이 어휘) | ✅ 완료 |
| (통합) | 둘이 하나로 — 실리콘 가속 백엔드가 같은 API 뒤로 | 테이프아웃 이후 | ⏸ 실리콘 관문 |

이 분리가 핵심이다: **소프트웨어 백엔드는 "느리지만 진짜 텍스트"라고 명확히 라벨**하고,
FPGA는 "빠르지만 토이"라고 라벨한다. 둘 다 진짜이고, 둘 다 과대표현이 아니다.
설계 파트너에게 "지금 보여줄 것"이 생긴다.

## 3. v0.1 제품 표면 (개인 파워유저) — 두 층

**데이터 플레인 (대화) = 생태계 클라이언트, 자체 안 지음**
- **챗 + 코드**: OpenAI 호환 `/v1/chat/completions`(이미 `host/aipu_server.py`에 껍데기
  존재) → Open WebUI·Continue.dev·Cursor 등 표준 클라이언트가 바로 붙음.
- **오프라인**: 호스트 로컬에서 완결, 밖으로 나가는 경로 없음.
- **캐싱**: 호스트 프리픽스 캐시 + 턴 간 KV 재사용(무조건).

**컨트롤 플레인 (기기 관리) = 자체 경량 웹 콘솔** — AIPU 고유, 생태계가 못 함:
- **헬스 대시보드**: `PROTO_CTX` 텔레메트리 리드백(tok/s·수락률·스톨·전원/발열)을
  실시간 상태로. "잘 돌고 있나"를 한눈에.
- **설정**: 전원/eco 모드, 컨텍스트 수, 캐싱, 로드할 모델 선택.
- **프로비저닝/업데이트**: 올라간 모델·A/B 슬롯 상태·467GB 로딩 진행률.
- **기능 제어 자리(성장)**: RAG 인덱스 관리·문서 추가·기능 토글은 **조직 비치헤드 때**
  이 콘솔이 자라나는 자리로 예약(사이트의 "GUI 튜닝" 비전이 여기로 연결).
- 형태: 호스트 서버(또는 박스)가 `aipu.local` 등에 서빙하는 **작은 상태+설정 화면**
  (풀 챗앱 아님 — 부담 적음).

## 4. 단계별 로드맵

- **Stage 0 — 완료**: 검증된 RTL, FPGA 핏(46.5MHz), 실 프로비저너, 락인 3개 파운데이션
  (부팅 무결성·가중치 ECC·프로토콜 ctx-id), 사이클 측정.
- **Stage 1 — v0.1 (일부 완료 · 2026-07)**: OpenAI API + **소프트웨어 풀모델 백엔드**로
  표준 클라이언트에서 **실제 텍스트**. 두 백엔드:
  **(주) Modal** — `tools/modal_glm_server.py`가 **실제 GLM 패밀리(GLM-4.5-Air-FP8,
  측정에 쓴 모델)**를 vLLM으로 Modal GPU 클라우드(2×H100, 유휴 시 0으로 스케일)에
  서빙, `host/aipu_modal_backend.py`가 프록시. **(대안) llama** — 로컬 작은 GGUF,
  오프라인·무비용. 그리고 **자체 경량 관리 콘솔**(`GET /console`: 헬스/텔레메트리 +
  설정 + `provision_image.py` 매니페스트를 읽는 프로비저닝 패널). 호스트 측 배관
  라이브 검증 완료(Modal 프록시 체인 + llama 실 GGUF). 남은 v0.1: 프리픽스/KV 캐시
  (v0.2). *(소프트웨어 백엔드는 가속기가 아니라 경험 증명 — 실리콘이 같은 API 뒤에서
  이 백엔드를 대체하는 게 Stage 3.)*
- **Stage 2 — 멀티컨텍스트 & 세션**: ctx-id 코어 스케줄러 + 호스트 세션 매니저 +
  프리픽스/KV 캐시 + 히스토리 영속성.
- **Stage 3 — 실리콘**: 테이프아웃(진짜 관문) → 실 가속 백엔드가 같은 API 뒤로.
- **Stage 4 — 조직 비치헤드**: RAG + 임베디드 GUI + 시각화(별도 시장 추구 시).

## 5. 비-목표 (개인 비치헤드 v0.1에서 안 함)

**자체 챗 GUI**(생태계 클라이언트로 대체), RAG, 지식그래프 시각화, 다중 사용자
인증 — 전부 **조직 비치헤드**로 연기. *(단, 자체 **관리 콘솔**은 v0.1에 포함 —
챗 GUI와 다른 것이며, 기기 관리는 생태계가 못 하므로 필수. §3 참조.)*
USB4 전송(실리콘 인터페이스; v0.1 소프트웨어는 로컬 HTTP).

## 6. 이미 깔린 기반 (기획이 공중에 뜨지 않는 이유)

- OpenAI 호환 API 서버 껍데기 + 토크나이저: `host/aipu_server.py`, `host/aipu_tokenizer.py`.
- 실 GGUF → 이미지 프로비저너: `tools/provision_image.py`(실 GGUF 2개로 증명).
- 프로토콜 ctx-id + 텔레메트리 리드백: `glm_q4k_system_cdc.v`(`PROTO_CTX`).
- 부팅 무결성·가중치 ECC 파운데이션: `boot_loader.v`(`INTEGRITY`), `weight_loader_q4k.v`(`WEIGHT_ECC`).
- 갭 추적: [`USAGE_GAPS.md`](USAGE_GAPS.md).

## 7. 오너가 확인할 것

위 D2–D9의 추천 기본값. 특히:
- **D2** — 소프트웨어 백엔드를 "데모"로 받아들이나, 아니면 데모가 반드시 하드웨어에서
  돌아야 하나? (후자면 v0.1이 테이프아웃까지 밀림.)
- **D3** — 챗은 생태계 클라이언트 + 기기 관리는 자체 경량 콘솔, 이 분리에 동의하나?
  (챗 GUI는 안 짓고, 관리 콘솔은 v0.1에 포함.)
- **D7/D8** — 프로비저닝 방식과 A/B 업데이트 정책.

확인/수정 주면 그 결정대로 Stage 1(v0.1) 실제 구현 — 소프트웨어 백엔드를 API에
연결해 표준 클라이언트로 진짜 텍스트가 나오게 — 부터 착수한다.
