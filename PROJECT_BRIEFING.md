## 1. What this project is

This is a synthesizable Verilog inference accelerator built for exactly one published model ‚Äî **GLM-5.2** (`GlmMoeDsaForCausalLM`), a 753B-parameter Mixture-of-Experts transformer with ~40B active parameters per token ‚Äî and nothing else. Its defining thesis is **Q4_K-native, GGUF-native local inference**: the target weight store is the published **`unsloth/GLM-5.2-GGUF : UD-Q4_K_XL`** (467 GB, ~38% smaller than the 753 GB FP8 checkpoint), and the ggml **Q4_K** block format (4-bit codes + 6-bit sub-block scales + fp16 super-block scale/min, per `get_scale_min_k4`) is implemented in hardware so the datapath consumes GGUF Q4_K weights directly. The RTL is elaborated as a **small-but-faithful slice** (e.g. `MODEL_DIM=128`, 6 layers, 8 experts, `VOCAB=256`) in which every real GLM-5.2 dimension (6144 hidden, 64 heads, nope192/rope64, kv_lora512, 256 experts top-8, index_topk 2048, 78 layers, vocab 154880, 1M context) is a *parameter*, not a hardcode ‚Äî so the slice exercises the identical control/dataflow paths as full scale, and the same RTL **elaborates clean at the real 753B UD-Q4_K_XL shape** (`test/full_config_elab_wrap.v`, `configs/full_glm52.vh`). The work is split into a **prototype branch (frozen at `47fb7f8`)** that answered "does the architecture work and how fast?" and a **product track on `main`** ‚Äî where `main` develops exactly one thing: the Q4_K accelerator at **rung ‚ëÝ (FPGA prove-it)**, the offline single-user demo proving the full 753B-family GLM runs on real low-end FPGA silicon; the funded custom board (rung ‚ë°) and volume SoC/ASIC (rung ‚ë¢) are documented roadmap (`docs/HARDWARE_LADDER.md`), not code developed in `main` today. The old 5-stage scalar TPU-v2 core is **removed from the repo** (off the product path), retained only in git history.

**Prior FP8 track (preserved, not current).** This project was migrated FP8 ‚Üí GGML Q4_K. The FP8 datacenter-native datapath is **preserved on branch `fp8` + tag `fp8-verified-baseline`** and is referenced here only as the prior baseline ‚Äî it is **not** what `main` develops. See `docs/Q4K_RETARGET.md`.

**Honest scope of the moat (read this first ‚Äî it is deliberately narrow; refreshed 2026-07).** The Q4_K GEMM core is **bit-exact to an independent ggml-Q4_K reference** (`tools/q4k_ref.py`, the team's own Python reimplementation of ggml's `dequantize_row_q4_K`), verified at a small-but-faithful slice ‚Äî and that reimplementation is now **proven bitwise-equal to real GGUF bytes at the dequant layer** (376,586,096 weights ‚Äî Q4_K/Q6_K/Q8_0, two real published GGUFs ‚Äî vs llama.cpp's own `dequantize_row_*`, all bit-identical ‚Äî `docs/GGUF_CROSSCHECK.md`). The datapath consumes the full **mixed-type** UD-Q4_K_XL mix (Q6_K/Q8_0/F16 RTL consumers, `make mixedtype`), and the assembled forward pass has an **end-to-end numeric golden** (`make model-q4k`, 1155/1155 bit-exact vs the assembled numpy reference). Still honest and open: **llama.cpp whole-runtime numeric equality is out-of-contract** (attention/accumulation orders differ by design), and the 467 GB GLM checkpoint has not been run end-to-end. Every tok/s, cost, and LOI figure is **[EST]/[PENDING]** ‚Äî the FPGA fit is now **MEASURED** (Vivado XCKU3P, `fpga/results/`).

## 2. Architecture at a glance

Token forward path (Q4_K datapath capstone shown; the bf16 golden track mirrors it at `PE_M=1`):

```
      ‚îî‚îÄ for l in 0..L-1 ‚îÄ‚îÄ glm_decoder_block_q4k (pre-norm block):

                              ‚îÇ dsa_indexer‚Üítopk_select (top-2048)  kv_cache_pager (latent ring‚ÜíNVMe)
                              ‚îÇ rope_interleave_unit (Œ∏=8e6)         glm_softmax  (q¬∑K score = bf16)
           rmsnorm_unit ‚îÄ‚ñ∂ FFN ‚îÄ‚ñ∂ (+residual)

                              ‚îî MoE  (l‚â•3):   moe_router_q4k ‚Üí [union-skip loop in glm_decoder_block_q4k] ‚Üí swiglu_expert_q4k

      ‚îî‚îÄ [spec] mtp_head_q4k drafts t+2 (uses h_state) ‚îÄ‚ñ∂ spec_decode_seq accept/reject
```

- **(a) Q4_K / float arithmetic core** ‚Äî `q4k.vh` (exact ggml Q4_K fp16‚Üífp32 decode + `get_scale_min_k4` scale/min unpack), `glm_fp.vh` (bf16‚Üîfp32, fp32 mul/add/rsqrt golden), `glm_fp_pipe.v` (0-ULP registered pipelines, LAT 2/5/7/24/46), `glm_matmul_q4k.v` (Q4_K block-scaled GEMM: dequantize exactly per ggml, then fp32 MAC). *The single source of numeric truth for the quant datapath.*
- **(b) GLM operators** ‚Äî `rmsnorm_unit`, `rope_interleave_unit`, `glm_act` (SiLU/sigmoid), `glm_softmax`, `topk_select`, `swiglu_expert(_q4k)`. *bf16-store / fp32-compute pointwise + reduction ops.*
- **(c) Attention & MoE** ‚Äî `mla_attn(_q4k)` (latent attention), `dsa_indexer`+`kv_cache_pager` (sparse top-K + latent-KV paging), `moe_router(_q4k)`+`expert_cache_pf` (route 256‚Üítop-8, stream experts; the expert-union-skip batching is folded inline into `glm_decoder_block_q4k`, not a standalone module). *The two compute-heavy blocks.*
- **(d) Decoder block & full model** ‚Äî `glm_decoder_block(_q4k)` (one pre-norm layer), `glm_model(_q4k)` (embed ‚Üí L layers ‚Üí final norm ‚Üí LM head ‚Üí argmax). *Pure orchestrators; no arithmetic of their own.*
- **(e) Speculative decoding / MTP** ‚Äî `mtp_head(_q4k)` (t+2 draft head, exposes `h_mtp`), `spec_decode_seq` (accept/reject), `spec_decode_top`/`spec_batched_top` (K=1 verify and K+1 batch loops), `spec_chain_top` (K-step MTP-chain draft loop), `sampler`.
- **(f) Memory/streaming** ‚Äî `flash_xbar`/`ddr5_xbar` (banked N-channel read fabrics), `weight_loader_q4k`/`boot_loader` (DMA), `weight_decomp`/`weight_decomp2` (streaming lossless byte decompressors), `kv_cache_pager`. *Holds the model on the NVMe SSD, streams the per-token working set.*
- **(g) SoC top-levels & CDC** ‚Äî `glm_q4k_soc` ‚Üí `glm_q4k_system` (adds ddr5_xbar + weight_loader_q4k) ‚Üí `glm_q4k_system_cdc` (two-clock host/core), on `cdc_async_fifo` + `reset_sync`.
- **(h) Productization/DFT** ‚Äî `ecc_secded`+`ecc_mem_wrap` (SECDED), `kv_ecc_ring` (lane-SECDED KV ring), `mbist_ctrl` (March C-), `icg_cell`+`clk_en_ctrl`+`clk_gate_cluster` (glitch-free gating), `clk_throttle` (DVFS prescaler), `reset_sync`.

## 3. Subsystem deep-dive

**1. Q4_K / float arithmetic foundation.** The canonical numeric core: `q4k.vh` implements ggml's Q4_K decode ‚Äî fp16‚Üífp32 super-block scale/min plus the packed 6-bit sub-block scale/min unpack (`get_scale_min_k4`) ‚Äî and `glm_matmul_q4k.v` is the block-scaled GEMM that dequantizes each 4-bit code exactly to fp32 and accumulates via the fp32 MAC path. **This is the one true bit-exact datapath result**: `glm_matmul_q4k` matches the ggml-Q4_K reference `tools/q4k_ref.py` (a Python reimplementation of `dequantize_row_q4_K`) **bit-for-bit** (`make q4k`: `ALL 160 TESTS PASSED (40 tiles, bit-exact vs ggml Q4_K golden)`), and the primitives match too (`q4k_prim`: `ALL 18 TESTS PASSED`). `glm_fp.vh` is the combinational bf16/fp32 golden (mul/add with RNE+FTZ, Quake-seed `fp32_rsqrt`); `glm_fp_pipe.v` provides 0-ULP registered retimings (`fp32_add_pipe` LAT5, `fp32_mac_pipe` LAT7, `fp32_exp_pipe` LAT46), latencies from `glm_fp_pipe_lat.vh`. **Honest scope (refreshed):** bit-exactness holds against *our* ggml reimplementation at the slice ‚Äî and that reimplementation's dequant layer is now **sealed against real GGUF bytes** (`docs/GGUF_CROSSCHECK.md`); the Q6_K/Q8_0/F16 tensors of the dynamic mix now **have RTL consumers** (`q4k_mixed.vh` + per-column `w_type` routing, `make mixedtype`: 32/32 + loader-mixed 192/192). What remains out of scope: llama.cpp's **whole-runtime** arithmetic (accumulation/attention order ‚Äî out-of-contract).

**2. Elementwise/normalization operators.** All non-GEMM pointwise/reduction ops under the bf16-store/fp32-compute contract: `rmsnorm_unit` (fp32 Œ£x¬≤ reduce ‚Üí Quake rsqrt ‚Üí √óŒ≥), `rope_interleave_unit` (a "lean rebuild" replacing a 128-bit modulo + 16-iter CORDIC with a Q48 turn-fraction ROM + fp32 Taylor trig, Œ∏=8e6, ~10.5√ó smaller), `glm_act` (sigmoid and SiLU via range-reduced exp), `glm_softmax` (numerically-stable serial single-add accumulator), `topk_select` (shared by the MoE router and DSA indexer), and `swiglu_expert(_q4k)`. Each has a dedicated bench with an *independent* fp64 golden (glm_act sigmoid worst-err 1.13e-7, SiLU bit-exact). These are the **generic bf16/fp32 twins** that the assembled-model TBs also build against (see ¬ß4 and the assembled-golden gap).

**3. Attention (MLA + DSA + KV pager).** `mla_attn(_q4k)` is a 14-state orchestrator computing one query token's Multi-head Latent Attention per decode step: it low-rank-projects Q, caches *only* the compressed latent `[c_kv|k_rope]` (576 elts/token real, vs ~94 GB of full KV at 1M context), and reconstructs per-head K/V on the fly via `W_uk`/`W_uv`. The Q4_K sibling routes the 7 large weight projections through `glm_matmul_q4k` while keeping RMSNorm/RoPE/the activation√óactivation q¬∑K score/softmax/context in bf16, and adds `PE_M` query-row batching plus per-row RoPE position (`PER_ROW_POS`), per-row causal extent (`PER_ROW_SLEN`), and per-row *sequence* (`PER_ROW_SEQ`) ‚Äî so each batched row can be a **different sequence** attending its own KV window while sharing the query-side weight fetch ‚Äî all safe-defaulting to 0. `dsa_indexer` caps attention FLOPs at ~134 MFLOP/query by scoring a cheap index vector through one software-pipelined shared `fp32_mac_pipe` and selecting top-2048 keys; `kv_cache_pager` is a power-of-two latent ring with NVMe overflow, extended to `NSEQ` independent per-sequence windows (NSEQ=1 byte-identical). **(Closed ‚Äî former GAP #1):** the assembled `mla_attn_q4k` attention (including the `1/sqrt(d_head)` softmax scale) is now covered by the assembled end-to-end numeric golden ‚Äî `make model-q4k` asserts the full `glm_model_q4k` forward (logits+argmax+h_state) byte-identical to the assembled numpy reference `tools/glm_model_q4k_ref.py` (1155/1155, commit b058f6f).

**4. MoE routing + expert cache/streaming.** `moe_router_q4k` scores 256 experts via a Q4_K gate GEMV then a bf16 sigmoid‚Üítop-8‚Üí**renormalize-then-scale (√ó2.5)** tail (the load-bearing, non-reorderable GLM/DeepSeek-v3 order ‚Äî routed weights sum to 2.5, not 1). Verified as **structural/functional invariants** (`make q4k`: `moe_router_q4k ‚Ä¶ ALL 40 TESTS PASSED (renorm Sum=SCALE, valid idx ‚Ä¶)`) ‚Äî invariants, **not** a numeric golden. The expert-union-skip loop ‚Äî **folded inline into `glm_decoder_block_q4k`** (the PE_M>1 `esc` scan cursor, "mirrors batched_moe.v S_SCAN"; the standalone `batched_moe.v` module was removed from main) ‚Äî is the throughput lever: it scans experts 0..N-1 and fetches each *distinct* union member once, so B token rows share one expert fetch, accumulating in ascending-index order for byte-exact batch-invariance. `expert_cache_ctrl`/`expert_cache_pf` implement an exact move-to-front LRU (with optional freq-aware/prefetch supersets that constant-fold to byte-identical fallbacks). The **predictor-prefetch stack is a documented honest no-op** at real cache size (routing entropy-capped) ‚Äî kept as verified RTL, marked do-not-re-propose.

**5. Decoder-block & full-model orchestrators.** `glm_decoder_block(_q4k)` sequences one pre-norm block (`h = x+attn(rmsnorm(x)); y = h+FFN(rmsnorm(h))`) reusing one shared `rmsnorm_unit` and time-multiplexing one `swiglu_expert` over TOPK+1 experts; `glm_model(_q4k)` walks one decoder-block instance across L layers (first 3 dense, rest MoE), applies final norm, runs the LM-head GEMV, and argmaxes. These reimplement *no arithmetic* ‚Äî correctness reduces to wiring plus already-proven leaves. `glm_model_q4k` exposes `h_state` (the additive hook feeding MTP). **The assembled Q4_K model now has its own numeric golden** ‚Äî `make model-q4k`, 1155/1155 bit-exact vs `tools/glm_model_q4k_ref.py` (see ¬ß4).

**6. MTP & speculative decoding.** The decode-loop throughput lever. `mtp_head(_q4k)` is a pure orchestrator that from the main hidden state `h_t` and the just-predicted token drafts the t+2 token (3 serial RMSNorms ‚Üí `W_proj` ‚Üí one `glm_decoder_block` ‚Üí LM head ‚Üí argmax; `mtp_head_q4k` exposes the `h_mtp` chain state ‚Äî the bf16 `mtp_head.v` does **not**). `spec_decode_seq` is the accept/reject controller; crucially it commits *only* the model's own argmaxes, so the committed stream is always a prefix of greedy decoding ‚Äî **speculation changes speed, never output**. This `spec==greedy` safety property is what the spec loops prove:
- `spec_decode_top` (in `make unittests`): `SPEC-DECODE EXACTNESS: committed verified-token sequence == greedy` / `ALL 18 TESTS PASSED`. **Caveat:** the greedy "golden" `u_ref` is *itself a standalone `glm_model_q4k` sharing the same weight ROMs as the DUT* ‚Äî this proves spec-decode == plain greedy argmax of the same model (a real lossless-speculation property), i.e. **DUT-vs-DUT self-consistency, not a numeric golden.**
- `spec_batched_top` and `spec_chain_top` (in `make spec-slow`, minutes-long, kept out of `unittests`): same `spec==greedy` property at larger K against an **independent** `glm_model_q4k` reference. `spec_chain_top` chains the single `mtp_head_q4k` recurrently on `h_mtp` to mint K drafts (self-draft K_eff ~1.7‚Äì2.2). Same DUT-vs-DUT caveat.

**7. Memory/weight-streaming fabric.** Realizes the tiered-storage strategy: the whole model lives on a 1‚Äì4 TB NVMe SSD (M.2 / PCIe), the fixed hot working set is copied to multi-channel DDR at power-up (`boot_loader`, gated by a registered `done`), and data-dependent experts are demand-streamed. The DDR tier is **rung-dependent** ‚Äî DDR4 (~4 ch) on the prove-it FPGA, DDR5 multi-channel or HBM on the funded board (`docs/HARDWARE_LADDER.md`). `flash_xbar`/`ddr5_xbar` are address-striped banked read fabrics; `flash_xbar` adds the `QDEPTH` outstanding-request knob for Little's-law latency hiding ‚Äî its read-request abstraction (address ‚Üí weight bytes) is medium-agnostic and in the product fronts an NVMe/PCIe host controller (a labeled placeholder). `weight_decomp`/`weight_decomp2` are streaming lossless canonical-Huffman decompressors (order-0 1.34√ó; order-1 context-modeled ~1.42√ó **[EST]**) operating on quant bytes as opaque symbols. `ckpt_pack_q4k.py` bridges the published GGUF to the exact word-addressed image `weight_loader_q4k.v` reads; `flash_layout.py` does offline expert‚Üíchannel placement. Every module has a self-checking TB; real PHYs are vendor-IP stubs modeled as latency memories.

**8. SoC top-levels & CDC.** Pure wiring/integration, no math reimplemented. `glm_q4k_soc` places `glm_model_q4k` with `expert_cache_pf` + `kv_cache_pager` over one demand-arbitrated NVMe channel; `glm_q4k_soc_ms` is its **batched multi-sequence** sibling ‚Äî `glm_model_q4k` at `PE_M=B` with `PER_ROW_SEQ=1`, a real `NSEQ`-window `kv_cache_pager` + `expert_cache_pf`, and a host FSM that prefills B sequences into a top-owned per-layer KV store (`kv_mem`), runs ONE batched forward, and commits B tokens; its multi-step **continuous-batching decode loop** (`N_STEPS>1`) streams N tokens/seq. `glm_q4k_system` adds `ddr5_xbar` + `weight_loader_q4k`; `glm_q4k_system_cdc` wraps the single-clock box in a two-clock host(USB)/core shell where **not one raw multi-bit value crosses domains directly** ‚Äî multi-bit through gray-pointer `cdc_async_fifo`s, level signals through 2-FF syncs, pulses via toggle+edge-detect. **PHY-closure loopback (proven 2026-07):** the `LOOPBACK=1` path physically routes the die's attention-weight Q4_K code lanes OUT as a banked `ddr5_xbar` read and back IN through the fabric; `make loopback` (`test/glm_q4k_loopback_tb.v`) proves the committed stream is BIT-EXACT to a standalone `glm_model_q4k` reference over a 4-token decode (7168 marked reads round-tripped), and a `-DLBINJECT` build that corrupts the fed-back lanes correctly FAILS. This closes the digital feed-back round-trip for the aw family; extending it to every weight family through the real vendor PHY IP is the downstream (rung-‚ë¢) step.

**9. P2 reliability/DFT/power.** The productization layer: `ecc_secded` (parameterized extended-Hamming SECDED, default (72,64), that deliberately does *not* correct double-bit errors), `ecc_mem_wrap` (ECC RAM with scrub-write-back + sticky `serr`/`derr` + back-door fault-injection), `kv_ecc_ring` (lane-SECDED KV ring, unit + `kv_cache_pager_ecc_fv` formal), `mbist_ctrl` (March C-, 10N), `icg_cell`+`clk_en_ctrl`+`clk_gate_cluster` (glitch-free gating, `req=1 ‚áí clk_en=1` safety invariant), `clk_throttle` (peak-power DVFS prescaler; unit + BMC verified), and `reset_sync` (now wired at both host/core boundaries of `glm_q4k_system_cdc`). Each is unit-verified ‚Äî `ecc_secded_tb` is exhaustive over all 72 single-bit and C(72,2)=2556 double-bit flips. **On MBIST/ICG in the top (corrected 2026-07):** `icg_cell` is already realized ‚Äî the die-wide clock gate `die_clk = clk & die_en_lat` (`glm_q4k_system.v:1307-1311`) is that glitch-free pattern hand-coded, freezing `u_model` on stall; finer ICGs are synthesis-inferred. `mbist_ctrl` is the verified single-port March **reference**: the production resident stores (`kv_cache_pager.ring`, `mla_attn_q4k.vstore_mem`) are 2-port async, so hand-wiring one single-port engine in would be theatre ‚Äî real BIST is a per-macro collar the memory compiler emits (contract in [`docs/P2_MEMORY_MAP.md`](docs/P2_MEMORY_MAP.md) ¬ß4). The memory macros and their BIST collars are the physical-flow (rung-‚ë¢) step.

**10. Formal verification.** A read-only harness suite (`test/formal/`) that bounded-model-checks committed memory-system controllers via `yosys write_smt2` + `yosys-smtbmc -s z3`, with a k-induction subset upgrading chosen properties to unbounded proofs. `make formal` (BMC, all PASS): `ddr5_xbar K=12`, `flash_xbar K=12`, `boot_loader K=16`, `clk_throttle K=16`, `spec_decode_seq K=20`, `kv_cache_pager K=16`, `expert_cache_pf K=20` (proven with prefetch **disabled**, `PF_ENABLE=0`), `kv_cache_pager_ecc_fv K=12` ‚Äî **7 controllers + 1 ECC-ring harness**. `make formal-ind` (k-induction, UNBOUNDED, all PROVEN): `boot_loader K=8`, `kv_cache_pager K=16`, `spec_decode_seq K=2`, `ddr5_xbar K=12`, `flash_xbar K=3`. Properties: routing/one-hot exclusivity, FIFO no-overflow/underflow, token-accounting equality, ECC encode-decode identity, done-gates, counter monotonicity. The load-bearing tooling note: `async2sync; chformal -lower` is mandatory before `write_smt2` or yosys 0.66 silently drops all `$check` cells and every proof passes *vacuously* ‚Äî non-vacuity is re-guarded by `grep -c assert` and PROBE_* reachability asserts. **Scope: NO formal proof touches the Q4_K numeric datapath** ‚Äî it is memory-system / control-plane safety only, BOUNDED for BMC (first K cycles), UNBOUNDED only for the k-induction subset.

**11. Build/test infra & Q4_K golden methodology.** `make all = unittests synth-glm formal`. `make unittests` builds+runs ~35 per-unit iverilog TBs (each with an in-TB fp64/real X-aware golden, gated on the exact string `ALL N TESTS PASSED`) plus the Q4_K subset (`make q4k`) and `spec_decode_top`. Separately: `lint` (verilator -Wall), `synth-glm` (whole-chip structural gate), `formal`/`formal-ind`, `cdc`, `coverage`, `spec-slow`, `host-test`. The Q4_K contract is proven as **bit-exact identity to `tools/q4k_ref.py`** (a faithful Python reimplementation of ggml `dequantize_row_q4_K`), the key being that an exact Q4_K dequant decomposes to scale √ó (code ‚àí min) letting a host-scale reference reproduce the hardware GEMM to the last bit. **Uneven by design:** only `glm_matmul_q4k` and `q4k_prim` are bit-exact-vs-ggml; `swiglu_expert_q4k` is self-labeled **functional** (`ALL 240 TESTS PASSED, functional vs Q4_K golden`); `moe_router_q4k` is **structural invariants**. The stance is honest: they prove the RTL matches a faithful ggml-Q4_K engine bit-for-bit **at the GEMM core**, not that the assembled model matches llama.cpp or the real GGUF.

**12. Project direction/status.** The governing docs (`docs/PRODUCT_ROADMAP.md`, `docs/ACCEL_GLM52.md`, `docs/Q4K_RETARGET.md`, `docs/Q4K_SYSTEM_PLAN.md`, `docs/HARDWARE_LADDER.md`) frame the prototype‚Üíproduct transition around the open gates ‚Äî of which the assembled-Q4_K numeric golden (`make model-q4k`), the mixed-type path (`make mixedtype`), and the FPGA fit (measured, `fpga/results/`) are now **closed**, leaving real-checkpoint whole-runtime validation and P2 closure ‚Äî and an NVMe/PCIe-bandwidth-bound perf thesis. Everything carries an honesty tag (**[EST]/[PENDING]** markers throughout the docs); measured no-ops (predictor-prefetch, deep lookahead) are reported rather than hidden. Recent commits closed batched-decode correctness gaps (per-position KV, per-row causal extent, the expert-union-skip MoE batching folded inline into `glm_decoder_block_q4k`), the multi-sequence batched-attention track (`PER_ROW_SEQ`, `glm_q4k_soc_ms` + its `N_STEPS` decode loop), and P2 reliability foundations.

## 4. Verification & correctness story

**Golden-model methodology.** Every *unit* is checked against an *independent* golden built inside its own testbench in fp64 (`real`) or bit-exact-Q4_K arithmetic. Benches are **X-aware** (any X/Z bit is a hard failure), compare against a bf16-ULP-scaled tolerance, and terminate with `ALL N TESTS PASSED` or `$fatal`. `make all` ‚Äî the GLM rung-‚ëÝ prove-it gate (`unittests synth-glm formal`) ‚Äî is a single binary CI signal.

**What is bit-exact vs functional vs structural (the honest tiering):**
- **Bit-exact vs ggml-Q4_K reference:** `glm_matmul_q4k` (160 tests), `q4k_prim` (18 tests) ‚Äî the pure-Q4_K GEMM core and its primitives.
- **Functional vs Q4_K golden (self-labeled, not bit-exact):** `swiglu_expert_q4k` (240 tests).
- **Structural/functional invariants (no numeric golden):** `moe_router_q4k` (40 tests).
- **Assembled Q4_K spec loop:** `spec_decode_top` (18 tests) ‚Äî `spec==greedy` **self-consistency** (DUT-vs-DUT, shared weight ROMs), *not* a numeric golden.
- **Assembled Q4_K numeric golden:** `make model-q4k` (1155 tests) ‚Äî the full `glm_model_q4k` forward bit-exact (logits+argmax+h_state) vs the assembled numpy reference `tools/glm_model_q4k_ref.py`.
- **Mixed-type path:** `make mixedtype` ‚Äî Q6_K/Q8_0/F16 dequant + `w_type` routing, bit-exact vs the ggml-reimpl goldens (32/32 + loader-mixed 192/192).

**GAP #1 ‚Äî CLOSED (assembled end-to-end numeric golden).** The assembled Q4_K path now has its own numeric gate: `make model-q4k` runs the full `glm_model_q4k` forward (embed ‚Üí L√ó(MLA+DSA+MoE) ‚Üí final norm ‚Üí LM head ‚Üí argmax) against the assembled numpy reference `tools/glm_model_q4k_ref.py` (which imports the same `q4k_ref.py` dequant) ‚Äî **1155/1155 bit-exact** on logits+argmax+h_state, including the `1/sqrt(d_head)` MLA softmax scale (commit b058f6f; `model-q4k-acthw` re-proves it with the ACT_HW knob). The per-unit assembled-model TBs (`glm_model_tb` etc.) still build the generic bf16/fp32 twins ‚Äî the Q4_K numeric gate is `model-q4k`. Caveat kept honest: the golden is still our **own reimplementation**, not llama.cpp's runtime.

**GAP #2 ‚Äî narrowed to the whole-runtime question.** (a) The goldens come from the team's own ggml reimplementation `tools/q4k_ref.py` ‚Äî whose **dequant layer is now proven bitwise-equal to real GGUF bytes** (376,586,096 weights ‚Äî Q4_K/Q6_K/Q8_0 across two real published GGUFs ‚Äî vs llama.cpp's own dequant ‚Äî `docs/GGUF_CROSSCHECK.md`; by transitivity the RTL dequant ‚â° the real files' ggml dequant). (b) The mixed-type Q6_K/Q8_0/F16 tensors now **have RTL consumers** (`make mixedtype`). What honestly remains: **llama.cpp whole-runtime numeric equality is out-of-contract** (attention/accumulation order differs by design), and the real 467 GB UD-Q4_K_XL has not been consumed end-to-end (see the real-checkpoint gate below).

**Make targets.** `unittests` (~35 unit TBs + `q4k` + `spec_decode_top`, self-bootstrapping via `route_trace.py --dump`), `q4k` (the four Q4_K TBs: 18/160/240/40), `synth-glm` (whole-chip elaboration + `check -assert`), `formal`/`formal-ind`, `spec-slow` (`spec_batched_top`/`spec_chain_top`), `cdc`, `coverage` (verilator line/toggle), `lint`, `host-test`.

**Real-checkpoint validation (OPEN ‚Äî the #1 fidelity gate).** There is **no** in-repo tool that runs the real 753B GLM against our datapath (the prior FP8 `modal_validate.py` was on the deleted track). Validating assembled-Q4_K token fidelity against the llama.cpp **runtime** / the real 467 GB GGUF is the **OPEN next step** ‚Äî its former prerequisites (the assembled numeric golden and the mixed-type path) are now closed, and the dequant layer is sealed against real GGUF bytes (`docs/GGUF_CROSSCHECK.md`); what remains needs a GPU/large-memory host.

## 5. Performance thesis

The machine is **NVMe/PCIe-bandwidth-bound, not compute-bound**: per token you must read cold routed-expert weights from the NVMe SSD plus the hot resident weights from fast DDR, and that read time dwarfs the math ‚Äî the compute die sits only ~20‚Äì25% utilized. The governing model is **tok/s ‚âà NVMe_BW / [(1‚àíh)¬∑footprint] ¬∑ K** and **J/token ‚âà bytes_moved √ó energy/bit**. Q4_K shrinks the store to **467 GB (~38% smaller than the 753 GB FP8 checkpoint)**; the canonical per-token byte constants are now derived in `docs/R3_APPLIANCE_SPEC.md` ¬ß2 ‚Äî **routed-expert stream ‚âà 14 GB/token** (75 MoE layers √ó top-8 √ó 37.75M params √ó ~0.62 B/param, growing √óU(K) under the spec-chain) plus **hot-set touch ‚âà 11 GB/token** (resident hot partition ~17 GB) ‚Äî still **[EST]** at the energy level (the old NAND-derived ~24‚Äì26√ó ratio does not transfer). Die-side work (fmax, formal proofs, DFT) improves cost/thermals/correctness but **does not move tok/s** until the NVMe read path is unblocked; the one die-side lever that does is `weight_decomp` (lossless byte-count reduction on the NVMe‚ÜíDDR refill).

**The projection [EST]** follows a **3-rung hardware ladder** set by memory bandwidth = IO pins + hard PHYs = budget (`docs/HARDWARE_LADDER.md`), all B=1 single-user on the same bit-exact RTL:
- **‚ëÝ prove-it FPGA (KU3P-class + DDR4), ~5‚Äì8 tok/s [EST]** ‚Äî now, proving it runs.
- **‚ë° funded custom board (DDR5/HBM), ~15‚Äì40 tok/s [EST]** ‚Äî after seed.
- **‚ë¢ volume ASIC, ~40+ tok/s [EST]** ‚Äî HBM stacks + many-channel controllers break the FPGA IO/PHY ceiling; NRE pays off only at volume. *(Updated 2026-07: the rung-‚ë¢ **primary** design point is now **512 GB LPDDR5X full residency** ‚Äî the whole ~467 GB checkpoint DRAM-resident (h=1 by construction), cold store one M.2 NVMe (boot-load ~70 s), design point **‚âà80 tok/s [measured-inputs EST]** (U(K) and the MTP accept rate r both GLM-family measured ‚Äî `docs/H_MEASUREMENT.md`; ~95 if GLM-5.2's deeper MTP hits its published accept depth); HBM stays the long-range ceiling ‚Äî see `docs/R3_APPLIANCE_SPEC.md` and the pivot section in `docs/HARDWARE_LADDER.md`.)*

The old flat ~25‚Äì40 was the funded-rung (‚ë°) number, not a near-term-cheap one. Levers that move the needle: multi-channel banking + deep outstanding queues, lossless decompression (order-0 1.34√ó ‚Üí order-1 ~1.42√ó **[EST]**), offline expert‚Üíchannel placement, speculative K, and ‚Äî for throughput serving only ‚Äî expert-grouped batching. **The honest no-op:** the confidence-thresholded predictor-prefetch stack is a measured no-op at realistic cache size (GLM's fine-grained routing is entropy-capped) ‚Äî kept as verified RTL, marked do-not-re-propose. **All system tok/s numbers are estimates** pending a real 753B routing trace and validated NVMe/PCIe bandwidth on the target board.

**The product is a local, single-user personal box** whose headline is **frontier AI that works with the ethernet unplugged** ‚Äî it runs the full 753B model **fully offline / air-gapped, so nothing leaves because there is no path out** (the audit is literally "does it still work with the cable pulled?"). Offline is framed as *capability*, not just a safeguard: (1) air-gap confidentiality, (2) disconnected-environment operation as a new market (SCIFs, defense forward ops, isolated OT/critical-infra, field/edge), (3) resilience/sovereignty (no vendor to rate-limit, deprecate, or cut you off). The moat is the *combination* ‚Äî offline + full-frontier 753B + appliance/seat price ‚Äî since offline **alone** is table-stakes for any local box. Honest caveats: the checkpoint is provisioned **once** (itself doable offline) and model updates are physical re-provisioning; aggregate/datacenter batching (per-user ~0.14 tok/s at B‚âà256) is a **non-target** analysis of the same silicon, not this product.
